Product positioning · A practical browser workspace for real debugging and validation

让 Claude 和 Codex 真正看见用户正在使用的浏览器。

ClaudeChrome puts Claude, Codex, and shell panes directly inside Chrome so teams can debug broken flows, validate releases, reproduce support tickets, and inspect real browser behavior without juggling separate terminals, tabs, and tools.

Use it when the bug only makes sense in the browser.

When a login loop, checkout regression, or front-end runtime error only becomes clear with the page in front of you, ClaudeChrome keeps the agent inside that context instead of forcing a manual relay between browser and terminal.

Useful for engineering, QA, support, and release validation.

Open a workspace, bind a pane to the exact tab you care about, inspect what the page is doing, and keep moving without losing the context that usually disappears the moment you switch tools.

Operator FAQ · Answer the adoption questions first

What people will ask before they try it.

Is ClaudeChrome mainly a debugging tool?

Debugging is the clearest entry point, but the product is equally useful for QA walkthroughs, support reproduction, release validation, and any workflow where the browser itself carries the important evidence.

Why not just use DevTools and a normal terminal side by side?

Because the value is not just having both tools open. It is keeping the agent session attached to the relevant browser tab so the investigation thread stays coherent while you move between inspection and action.

Do I need a full automation setup to benefit from it?

No. ClaudeChrome is useful earlier than that. It helps the moment you want a browser-aware agent workspace for real pages, real flows, and real validation work.

Where should screenshots and terminal recordings live?

Keep them versioned with the site. This landing page already reserves `src/ui/assets/*.png` for screenshots and `src/ui/assets/casts/*.cast` for asciinema recordings.
