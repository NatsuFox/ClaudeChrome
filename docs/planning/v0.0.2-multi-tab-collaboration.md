# Plan: ClaudeChrome v0.0.2 Multi-Tab Collaboration

## Context

This worktree is dedicated to `dev-0.0.2-multi-tab-collaboration`.

The current runtime already has the right lower-level shape for multi-tab collaboration:

- browser capture is stored by `tabId` in `native-host/src/context-store.ts`
- sessions keep a stable default `bindingTabId` in `native-host/src/session-manager.ts`
- the MCP and IPC layer currently forces every tool call through `sessionId -> bound tab` resolution in `native-host/src/main.ts`

That means the main architectural problem is not the collector layer. The problem is the tool-targeting policy.

## Goal

Ship a first-class multi-tab collaboration model where one Agent session can coordinate work across multiple tabs without losing a stable default context.

The design target is:

- every session still has a primary bound tab
- most browser tools can optionally target another tab for a single call via `tab_id`
- if `tab_id` is omitted, the tool still uses the bound tab
- explicit tools exist for `activate tab` and `bind tab`
- multi-agent and multi-pane work stays possible, but is no longer required for ordinary multi-tab workflows

This directly supports flows such as registration + temp-email + confirmation without forcing one Agent per tab.

## Product Decision

### 1. Keep binding as the default context, not as an exclusive restriction

The bound tab remains valuable because it gives every session a clear home context and safe default target.

But binding should no longer mean that a session is unable to inspect or operate on any other tab.

Recommended rule:

- `bound tab` = the session's default target when `tab_id` is not provided
- `tab_id` override = a one-call target override that does not change the session binding
- `browser__bind_tab` = explicit ownership change for the session default target

### 2. Support single-Agent cross-tab loops directly

For coordination-heavy workflows, the default should be:

- one Agent session
- one stable bound tab
- optional cross-tab reads and actions through `tab_id`
- explicit `activate` when user-visible focus matters
- explicit `bind` only when the session's home context truly changes

This avoids the cost and coordination fragility of mandatory multi-Agent orchestration.

### 3. Distinguish three classes of tab targeting

#### Class A: Session-scoped only

These should remain session-centric and not accept `tab_id` in v0.0.2:

- `browser__binding_status`
- `browser__status`
- `browser__capabilities`
- `browser__capture_stats`
- `browser__self_check`
- `browser__explain_unavailable`

These describe the current session and its default binding, not arbitrary tabs.

#### Class B: Cross-tab capable read/query tools

These should accept optional `tab_id` in v0.0.2:

- `browser__get_requests`
- `browser__get_request_detail`
- `browser__search_responses`
- `browser__get_console_logs`
- `browser__get_page_info`
- `browser__get_page_text`
- `browser__get_page_html`
- `browser__get_page_content`
- `browser__find_elements`
- `browser__evaluate_js`
- `browser__get_storage`
- `browser__get_selection`

#### Class C: Cross-tab capable action tools

These should accept optional `tab_id` in v0.0.2:

- `browser__navigate`
- `browser__reload`
- `browser__click`
- `browser__type`
- `browser__scroll`
- `browser__wait_for`

`browser__get_cookies` and `browser__screenshot` should be handled more conservatively:

- `browser__get_cookies` can be left unchanged for v0.0.2 because it is already mostly URL-driven rather than tab-driven
- `browser__screenshot` should not gain free-form cross-tab targeting in v0.0.2 because Chrome screenshot semantics are tied to the visible active tab in a window; it is safer to keep screenshot behavior conservative until a focused design is chosen

### 4. Add explicit orchestration tools

Add:

- `browser__activate_tab(tab_id)`
- `browser__bind_tab(tab_id)`

Semantics:

- `browser__activate_tab` changes Chrome focus to the requested tab but does not change session binding
- `browser__bind_tab` changes the session default bound tab and returns the updated binding snapshot

## API and response design

### Optional `tab_id` parameter

For tools in Class B and Class C, add:

- `tab_id?: number`

Resolution rule:

- if `tab_id` is provided, use that target for the call
- otherwise, use the session's bound tab

### Response metadata

For all cross-tab-capable tools, include target metadata in the result:

- `resolvedTabId`
- `usedBoundTab` boolean

This makes the runtime behavior auditable and reduces wrong-tab confusion.

### Error behavior

Add structured errors for:

- invalid `tab_id`
- unavailable target tab
- target tab not captured yet for store-backed query tools

For store-backed tools, error payloads should report both:

- the session id
- the requested or resolved tab id

## Implementation design

## 1. Host target-resolution layer in `native-host/src/main.ts`

Introduce shared helpers so both commands and store queries can resolve tab targets consistently.

Planned helpers:

- `getSessionContext(sessionId)`
- `resolveTargetTabId(context, params, options)`
- `resolveTargetTabContext(context, params, options)`
- `withoutTabTarget(params)` or equivalent lightweight param sanitation helper

The important change is:

- do not use `sessionManager.getBindingTabId(sessionId)` directly inside IPC command handling anymore
- instead, fetch the session context once and resolve the target tab from `params.tab_id ?? boundTabId`

This preserves binding while enabling one-call overrides.

## 2. Host-side mutation command for rebinding

Add a host-side command path for session mutations that do not belong in the extension browser-command dispatcher.

Recommended shape:

- extend IPC protocol with `kind: 'host_command'`
- add `bind_tab` as the first host command

Behavior:

- validate that the target tab exists or is at least known enough to bind safely
- call `sessionManager.bindSessionToTab(sessionId, tabId)`
- return updated binding metadata

This keeps binding changes in the host, where session ownership already lives.

## 3. MCP bridge changes in `native-host/src/mcp-stdio-bridge.ts`

### Add optional `tab_id` to targetable tools

For the selected tools, extend the zod schema with `tab_id: z.number().optional()`.

### Add `browser__activate_tab`

Expose as a browser command over the existing command channel.

### Add `browser__bind_tab`

Expose as a host command over the new host-command path.

### Update tool descriptions

All updated tool descriptions should say:

- bound tab is the default
- `tab_id` can override the target for this call

This must be explicit so Agent behavior is predictable.

## 4. Extension command dispatcher changes in `extension/command-dispatcher.ts`

Add browser-command support for:

- `activate_tab`

Recommended implementation:

- activate the provided tab
- return a summarized tab record similar to the current panel runtime message path

This should reuse the same Chrome tab activation semantics the panel already relies on.

No broad redesign of the service worker is required for v0.0.2.

## 5. Capability surface updates

Update `IMPLEMENTED_SESSION_TOOLS` and the capability families in `native-host/src/main.ts`.

Planned changes:

- add `browser__activate_tab`
- add `browser__bind_tab`
- update the `tabs` family to include the new tools
- add a small capability hint documenting default bound-tab targeting plus optional `tab_id` override

### Primary code changes

Modify:

- `scripts/test-commands.js`

Potentially modify if type wiring requires it:

- `extension/shared/types.ts`
- `extension/service-worker.ts`

### Expected non-changes in v0.0.2

Do not redesign in this branch:

- pane/workspace UI model
- session manager ownership model
- collector storage format in `context-store.ts`
- full workflow key-value store
- generic multi-Agent coordination primitives

Those can come later if still needed after single-Agent cross-tab targeting lands.

## Verification plan

## 1. Unit-level dispatcher coverage

Extend `scripts/test-dispatcher.js` to validate:

- `activate_tab` dispatch resolves correctly
- existing commands still behave the same

## 2. IPC integration coverage

Extend `scripts/test-commands.js` to validate:

- a session bound to tab A can send a command to tab B via `tab_id`
- websocket browser-command payloads carry the resolved target tab id rather than always the bound tab id
- `browser__bind_tab` updates the session binding
- subsequent commands without `tab_id` use the new bound tab

## 3. Live browser validation

Extend `scripts/test-live.js` to create at least two real tabs and validate the actual collaboration seam.

Recommended live assertions:

- create session bound to tab A
- discover tab B via `browser__list_tabs`
- call a read/query tool with `tab_id=tabB` and verify tab B content is returned
- call an action tool with `tab_id=tabB` and verify tab B changes without rebinding
- call `browser__activate_tab(tabB)` and verify active tab changes
- call `browser__bind_tab(tabB)` and verify `browser__binding_status` now reports tab B
- verify a follow-up tool call without `tab_id` now resolves to tab B

This is the most important proof for the feature.

## 4. Full repo validation

After implementation:

- run `npm test`
- run `npm run test:live`

## Risks and edge cases

### 1. Screenshot semantics

Chrome visible-tab capture is not naturally aligned with arbitrary background-tab targeting.

Plan for v0.0.2:

- leave screenshot conservative and bound-centric
- do not overextend it into the new targeting model without dedicated design

### 2. Query tools over uncaptured tabs

A tab may exist but not yet have page info, request capture, or console capture in the host store.

Plan:

- return clear `not_captured_yet` errors with resolved tab metadata
- do not silently pretend that the bound tab or another captured tab is equivalent

### 3. Wrong-tab mutations

Optional `tab_id` makes the system more powerful but also easier to misuse.

Mitigation:

- include `resolvedTabId` in results
- keep binding as the safe default
- keep `bind` explicit and separate from per-call overrides

## Recommended implementation sequence

1. Refactor `main.ts` to support shared target resolution for commands and queries.
2. Add host-command support for `bind_tab`.
3. Extend MCP tool schemas and descriptions in `mcp-stdio-bridge.ts`.
4. Add `activate_tab` to `command-dispatcher.ts`.
5. Update capability metadata and implemented tool lists.
6. Expand `test-dispatcher.js` and `test-commands.js`.
7. Expand `test-live.js` into a two-tab single-session collaboration proof.
8. Run `npm test` and `npm run test:live`.

## Deliverable definition for this branch

This branch is successful if:

- one Agent session can inspect and act on multiple tabs in a single loop using explicit `tab_id`
- the bound tab still remains the default when `tab_id` is omitted
- tab activation and tab rebinding are both explicit MCP tools
- automated tests prove the feature through both mocked integration and live browser validation

## Deferred follow-up after v0.0.2

Possible next tranche, intentionally deferred from this branch:

- workflow/session shared value store for OTPs, temp emails, and extracted artifacts
- richer UI affordances for cross-tab targeting inside the side panel
- conservative `tab_id` support for cookies and screenshot after a dedicated design pass
- policy knobs for requiring explicit confirmation on cross-tab mutating tools
